The composition models the Ironic provision state machine as one custom resource per state:
- A Node CR (synced to Ironic by KOG / rest-dynamic-controller).
- A NodeConfiguration CR (Ironic endpoint headers).
- One NodeProvision CR per transition, rendered ONLY for the node's current provision_state
  (read live via the Helm `lookup` function): enroll -> manage, manageable -> provide,
  available -> active(deploy). Each NodeProvision fires PUT /v1/nodes/{node}/states/provision
  once (via the ironic-node-provision RestDefinition).

composition-dynamic-controller re-evaluates `lookup` on every reconcile, so it renders the
next transition and prunes the previous one, walking enroll -> manageable -> available ->
active. Once the node is active, no transition is rendered (terminal).

Watch:
  kubectl get nodes.baremetal.ogen.krateo.io {{ .Values.nodeName }} -o jsonpath='{.status.provision_state}'
  kubectl get nodeprovisions.baremetal.ogen.krateo.io -l app.kubernetes.io/instance={{ .Release.Name }}
